# MVP 范围

## 目标

第一版实现必须支持从全新开局到第 1 章结束的完整可玩流程。

目标玩家流程：

```text
新游戏
  -> 序章教程战 1：玲纱单人
  -> 序章剧情：灼夜加入
  -> 序章教程战 2：玲纱 + 灼夜
  -> 第 1 章「银誓余烬 / 银誓要塞遗址」步进式路线
  -> 第 1 章固定灼夜大招解锁战
  -> 第 1 章 Boss：断誓之盾·莱恩哈特
  -> 第 1 章通关结算、存档、第 2 章占位解锁
```

只有当以上流程可以在不使用 debug 工具的情况下完整打通时，MVP 才算完成。

## 必须实现的章节

### 序章

必须实现：

- 线性流程，不提供路线选择。
- 教程战 1：只上场玲纱。
- 剧情过渡：灼夜加入队伍。
- 教程战 2：玲纱和灼夜共同上场。
- 教程提示可以简化，但玩家必须学会：
  - 普攻消耗 AS，并立即获得能量
  - 卡牌消耗公共能量
  - 多名角色共享一个牌库和一个能量池
  - 结束回合后进入敌方行动

### 第 1 章

必须实现：

- 第 1 章总计 13 步。
- 章节标题为「银誓余烬」，主地点为「银誓要塞遗址」。
- 第 13 步固定为第 1 章 Boss。
- 首次游玩第 1 章时，第 6 步固定为灼夜大招解锁战。
- 路线选择采用 `map-encounters.md` 中的每步二/三选一模式。
- 第 1 章通关后保存进度，并清晰结束 MVP 流程。

第一版不要求支持“通关后重复游玩第 1 章”的完整规则；如果存档系统自然支持，可以保留。

## 必须实现的可玩角色

MVP 必须可玩：

- 玲纱
- 灼夜

MVP 中锁定、不可玩：

- 初叶
- 瑠璃
- 影月

这些角色的数据可以为后续兼容先存在，但 MVP UI 不应允许玩家选择她们上场。

## 必须实现的卡牌

必须实现：

- 玲纱和灼夜的全部固有初始卡牌。
- 大招锁定和解锁机制。
- 玲纱大招在第 1 章 Boss 战开始时解锁。
- 灼夜大招在第 1 章第 6 步固定战斗结束后解锁。
- 共享牌库、手牌、弃牌堆、消耗堆。
- 每回合抽 3 张，手牌上限 8 张。
- 卡牌费用、目标校验、效果执行。

如果第 1 章精英奖励使用卡牌升级，则必须实现卡牌升级系统。若实现前仍未定义第 1 章卡牌升级触发内容，引擎应保留升级能力，但第 1 章内容可以暂不发放升级奖励。

## 必须实现的敌人

必须实现的敌人内容：

- 灰烬游魂
- 堕落骑士
- 断誓之盾·莱恩哈特

必须实现的敌人变体：

- 序章教程战 1 使用的弱敌。
- 序章教程战 2 使用的多敌人编队。
- 第 1 章普通战：使用灰烬游魂变体或群组。
- 第 1 章精英战：使用堕落骑士。
- 第 1 章 Boss 战：使用莱恩哈特三阶段 AI。
- 莱恩哈特二阶段召唤的弱化灰烬游魂。

敌人 AI 必须数据驱动，或至少隔离在清晰模块中，确保后续章节能新增敌人而不重写战斗主循环。

## 必须实现的战斗系统

必须实现：

- 公共能量池，默认能量上限为 8。
- ER 初始为 0。
- 普攻消耗 1 AS，并立即获得 1 点能量。
- 出牌不消耗 AS。
- 每回合刷新 AS。
- 第一回合抢攻机制。
- 命中 / 闪避判定。
- 暴击判定和暴击伤害。
- 伤害和治疗最终结果取整数。
- DEF 使用减法模型，伤害最低为 1。
- 护盾机制，包括护盾上限为最大 HP 的 200%。
- 流血、毒、虚弱、脆弱。
- `poison_detonate` 读取毒层造成伤害，但不清空毒层。
- `bleed_detonate` 消耗流血层。
- 若 MVP 卡牌或奖励会用到，则必须支持：嘲讽、吸血、抽牌、能量获得、AS 获得 / 重置、buff、卡牌变形、消耗 / Exhaust。
- 全部敌人被击败时胜利。
- 我方全部角色阵亡时失败。
- 每场战斗开始时，上场角色 HP 回满。

第一版战斗动画可以简化，但状态变化必须可见、可理解。

## 必须实现的 run 系统

必须实现：

- 新 run 创建。
- 序章完成后进入第 1 章。
- 第 1 章前段二选一、后段三选一的路线选择。
- 固定步骤覆盖：
  - 首次第 1 章第 6 步：灼夜大招解锁战
  - 第 13 步：Boss
- 节点类型：
  - 普通战斗
  - 精英战斗
  - 事件
  - 商店
  - Boss
- `node-modifiers.md` 中可在第 1 章出现的节点修正。
- 如果实现前已定义触发规则，则实现战后随机事件。
- 节点之间可以使用药水。
- 进入第 1 章节点前按规则选择队伍，最多 3 人。MVP 通常只有玲纱 + 灼夜。

## 必须实现的奖励

必须实现：

- 战后属性成长三选一。
- 若经济规则启用，则支持单个选项刷新。
- 金币奖励。
- 药水掉落奖励。
- 精英和 Boss 装备掉落。
- 第 1 章 Boss 奖励流程使用当前 Boss 奖励规则。
- 第 1 章通关结果界面。

奖励系统必须支持第 1 章会用到的全部稀有度，包括 Boss 奖励需要的稀有度。

## 必须实现的装备

必须实现的机制：

- 每名角色 3 个通用装备槽。
- 获得装备时立即决定是否装备。
- 装备后锁定。
- 装备槽满时提供替换流程。
- 装备属性加成。
- 如果 MVP 装备池使用装备附带卡牌或改牌效果，则必须支持对应机制。
- 如果第 1 章 Boss 奖励可能提供诅咒装备，则必须支持诅咒装备处理。
- 如果商店内容启用装备出售 / 回收，则必须支持对应流程。

必须实现的内容：

- 第 1 章商店装备池。
- 第 1 章精英掉落装备池。
- 灼夜特殊精英奖励装备池。
- 第 1 章 Boss 奖励装备池。

具体第 1 章装备池应在编写物品数据前补齐。

## 必须实现的药水

必须实现的机制：

- 全队共享药水背包，容量上限为 3。
- 药水只能在节点之间使用。
- 药水效果永久、不可撤销。
- 需要目标时，使用药水时选择目标角色。
- 背包满时，获得新药水需要选择丢弃旧药水或放弃新药水。
- 如果第 1 章 Boss 奖励或商店配置可能出现传说药水，则必须支持传说药水效果。

必须实现的内容：

- 第 1 章药水掉落池。
- 第 1 章商店药水池。
- Boss 药水奖励池。

商店不直接卖卡，也不提供直接删牌服务。复制、删除、改造卡牌通过药水效果间接实现，尤其是传说药水。

## 必须实现的商店

必须实现：

- 商店节点 UI。
- 购买药水。
- 购买装备。
- 如果当前规则启用，则支持装备 / 药水回收。
- 如果第 1 章可能出现永久 debuff，则支持永久 debuff 净化 / 修理。
- 花费金币手动存档。
- 如果第 1 章定义了具体情报效果，则支持情报购买。

明确不允许：

- 直接购买卡牌。
- 直接删牌服务。

## 必须实现的事件

必须实现：

- 事件节点框架。
- 事件选项和结果。
- 事件结果可以影响金币、药水、节点修正或其他 run 状态。

内容依赖：

- 第 1 章需要补齐具体事件列表，才能算内容完整。

## 必须实现的存档 / 进度

必须实现：

- 新 run 存档状态。
- 序章完成或进入第 1 章后的自动存档。
- 第 1 章 Boss 胜利后的自动存档。
- 商店手动存档。
- 继续已保存 run。
- run 失败结果。
- 第 1 章通关结果。

MVP 的局外成长可以非常轻量，只需要保存足够状态，让系统知道第 1 章已通关，并能在未来解锁第 2 章内容。

## 必须实现的 UI 界面

必须实现：

- 标题 / 开始 / 继续。
- 序章剧情对话。
- 战斗界面。
- 手牌和目标选择。
- 结束回合按钮。
- 战斗结果。
- 奖励选择。
- 药水背包和药水使用。
- 装备获得和替换。
- 路线选择。
- 事件节点。
- 商店。
- 存档 / 读档反馈。
- 第 1 章通关结果。
- run 失败结果。

缺少最终美术资源时可以使用占位资源，但布局应遵循 `docs/04-art/style-guide.md`。

## 明确不在第一版范围内

第一版不要求实现：

- 第 2-5 章可玩内容。
- 初叶、瑠璃、影月作为可玩角色。
- 完整长期局外成长树。
- 无尽模式。
- 每日挑战或种子分享系统。
- 最终生产级动画质量。
- 完整本地化管线。
- 移动端专项优化。
- 运营或商业化系统。

## 开发前必须继续补齐的规格

接下来应继续创建：

- `data-schema.md`
- `battle-state-machine.md`
- `effect-dsl.md`
- `run-flow.md`
- `acceptance-cases.md`

以下第 1 章规则已在现有设计文档中定义，后续写 `run-flow.md` 或数据表时应直接采用，不应重复重新设计：

- 节点出现权重：见 `docs/02-systems/map-encounters.md` 的“选项生成规则”。
  - 步 1-4：普通战 50% / 精英战 10% / 商店 10% / 事件 30%。
  - 步 5-8：普通战 40% / 精英战 20% / 商店 20% / 事件 20%。
  - 步 9-12：普通战 40% / 精英战 30% / 商店 10% / 事件 20%。
  - 步 1 至少 1 个普通战选项，步 12 至少 1 个商店选项，步 13 固定 Boss。
- 战斗药水掉落：见 `docs/02-systems/roguelike-run.md`。
  - 普通战：30% 掉落 1 瓶，普通 70% / 稀有 30%。
  - 精英战：30% 掉落 1 瓶，稀有 70% / 史诗 30%。
  - Boss：必定掉落 1 瓶，史诗 75% / 传说 25%。
- 装备品质分布：见 `docs/02-systems/equipment.md`。
  - 第 1 章精英战：稀有装备 100%。
  - 第 1 章灼夜特殊精英：保底史诗装备。
  - 第 1 章商店装备：普通 100%。
  - 第 1 章 Boss：史诗装备（保底）或诅咒装备二选一。
- 装备内容池：默认按装备品质列表取池。
  - 第 1 章商店装备池 = 全部普通装备。
  - 第 1 章普通精英装备池 = 全部稀有装备。
  - 第 1 章灼夜特殊精英装备池 = 全部史诗装备。
  - 第 1 章 Boss 选项 A = 全部史诗装备，选项 B = 全部诅咒装备。
  - 如果后续要做“每个精英固定掉落表”，必须在内容表中覆盖这个默认规则。
- 商店药水数量：见 `docs/02-systems/potions.md` 与 `docs/05-data-drafts/shop.json` 草案；每个商店随机刷新 1-3 瓶药水，旧草案中 tier 配置为 3 / 4 / 4 瓶。MVP 若未另行定义，使用 1-3 瓶随机刷新。
- 商店药水品质分布：第 1 章先按商店商品品质分布“普通 100%”执行，除非后续 `run-flow.md` 单独定义药水商店分布。

第 1 章内容实现前仍需要补齐：

- 序章敌人的精确属性和编队。
- 第 1 章第 6 步灼夜大招解锁战的敌人和奖励配置。
- 第 1 章普通战遭遇表。
- 第 1 章精英战遭遇表，以及精英是否触发卡牌升级。
- 第 1 章事件列表和事件结果。
- 第 1 章节点描述文本池。
- 第 1 章战后随机事件触发概率。
- 精确的存档槽位和覆盖规则。

## MVP 验收标准

满足以下条件时，MVP 视为通过：

- 玩家可以从新游戏开始，不借助开发工具进入第 1 章。
- 序章能教会普攻回能和出牌。
- 玩家可以在第 1 章进行路线选择。
- 普通战、精英战、商店、事件、固定剧情战、Boss 节点都能正常完成。
- 战斗失败会结束 run，并进入失败结果。
- Boss 胜利会结束第 1 章，并进入通关结果和保存进度。
- 玲纱和灼夜的大招解锁时机符合设计。
- 奖励、装备、药水、金币会影响后续第 1 章战斗。
- 存档和继续游戏可以恢复第 1 章进行中的 run。
- 不需要第 2 章内容也不会卡死流程；显示“未开放 / coming next”占位状态即可。
